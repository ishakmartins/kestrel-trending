Serving Kestrel Trending at open.lokanetra.dev/kestrel-trending

Updated plan: instead of the default ishakmartins.github.io/kestrel-trending URL, this sets up a dedicated hub subdomain — open.lokanetra.dev — that houses Kestrel Trending now and can absorb more open-source projects and open datasets later, each getting its own clean subpath.

1. Why this is possible in one step, not per-project

GitHub Pages has a rule made for exactly this: if you set a custom domain on your user site (the special repo named ishakmartins.github.io), every other public repo's Pages site under your account automatically inherits that same domain as a subpath — no per-project DNS or config needed. A project repo named kestrel-trending with Pages enabled and no custom domain of its own will simply appear at open.lokanetra.dev/kestrel-trending once the user site owns the domain.

Practically, this means:

  • One-time setup: point open.lokanetra.dev at your GitHub user site.
  • Zero setup per future project: any new public repo just needs Pages turned on and it shows up at open.lokanetra.dev/<repo-name> automatically.

This is still fully free — custom domains, HTTPS certificates (via Let's Encrypt), and Pages hosting all come at no cost on public repos.

3. Step-by-step

A. Domain and DNS (one-time, account level)

You do this
1. If you haven't already, create the special repo ishakmartins.github.io (public). This becomes your open-projects hub/landing page.
2. In GitHub account settings, verify ownership of lokanetra.dev before adding the custom domain to Pages — this is a security step (a TXT record) that prevents subdomain takeover. Docs: Verifying your custom domain.
3. In the ishakmartins.github.io repo → Settings → Pages → Custom domain, enter open.lokanetra.dev and save. GitHub will add a CNAME file to the repo automatically.
4. At your DNS provider for lokanetra.dev, add a CNAME record: host open → value ishakmartins.github.io. (Not the repo name — just the bare user-site domain.)
5. Back in Pages settings, wait for the DNS check to pass, then tick Enforce HTTPS (can take up to ~24h for the certificate, usually much faster).

4. Future considerations — growing open.lokanetra.dev

Design notes for later
  • Adding a new open project: create the repo, enable Pages from a branch/folder, leave its custom domain blank. It appears at open.lokanetra.dev/<repo-name> with no DNS work.
  • Hub landing page: keep ishakmartins.github.io's index.html as a manually (or Claude-)maintained directory of what's live under the subdomain — a natural place for short descriptions, last-updated timestamps, and links to each project's own README/data license.
  • Open data, not just dashboards: the same pattern works for raw datasets — a repo can publish a docs/ folder of CSVs/JSON with a small index page, reachable at open.lokanetra.dev/<dataset-repo>, no separate hosting needed.
  • Consistent look: if you want shared styling/branding across projects (so the hub feels like one coherent site rather than a list of disconnected demos), a small shared CSS/header snippet copied into each project's docs/ is enough — no build system required.
  • Repo growth: hourly commits from Kestrel Trending's workflow will keep growing that repo's git history. Not a functional problem, but worth revisiting (e.g. periodic squashing, or moving generated output to a separate branch) if the repo becomes unwieldy — this can be handled per-repo without affecting the hub domain.
  • Repo-level override: if any future project ever needs its own separate domain instead of a subpath, that's still possible — setting a custom domain directly on that repo overrides inheriting the hub's domain.
